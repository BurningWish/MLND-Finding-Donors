Transforming Skewed Continuous Features¶

A dataset may sometimes contain at least one feature whose values tend to lie near a single number, but will also have a non-trivial number of vastly larger or smaller values than that single number. Algorithms can be sensitive to such distributions of values and can underperform if the range is not properly normalized. With the census dataset two features fit this description: 'capital-gain' and 'capital-loss'.

Run the code cell below to plot a histogram of these two features. Note the range of the values present and how they are distributed.

For highly-skewed feature distributions such as 'capital-gain' and 'capital-loss', it is common practice to apply a logarithmic transformation on the data so that the very large and very small values do not negatively affect the performance of a learning algorithm. Using a logarithmic transformation significantly reduces the range of values caused by outliers. Care must be taken when applying this transformation however: The logarithm of 0 is undefined, so we must translate the values by a small amount above 0 to apply the the logarithm successfully.

Metrics and the Naive Predictor¶

CharityML, equipped with their research, knows individuals that make more than \$50,000 are most likely to donate to their charity. Because of this, *CharityML* is particularly interested in predicting who makes more than \$50,000 accurately. It would seem that using accuracy as a metric for evaluating a particular model's performace would be appropriate. Additionally, identifying someone that does not make more than \$50,000 as someone who does would be detrimental to *CharityML*, since they are looking to find individuals willing to donate. Therefore, a model's ability to precisely predict those that make more than \$50,000 is more important than the model's ability to recall those individuals. We can use F-beta score as a metric that considers both precision and recall:

$$ F_{\beta} = (1 + \beta^2) \cdot \frac{precision \cdot recall}{\left( \beta^2 \cdot precision \right) + recall} $$

In particular, when $\beta = 0.5$, more emphasis is placed on precision. This is called the F$_{0.5}$ score (or F-score for simplicity).

Looking at the distribution of classes (those who make at most \$50,000, and those who make more), it's clear most individuals do not make more than \$50,000. This can greatly affect accuracy, since we could simply say "this person does not make more than \$50,000" and generally be right, without ever looking at the data! Making such a statement would be called naive, since we have not considered any information to substantiate the claim. It is always important to consider the naive prediction for your data, to help establish a benchmark for whether a model is performing well. That been said, using that prediction would be pointless: If we predicted all people made less than \$50,000, CharityML would identify no one as donors.

Note: Recap of accuracy, precision, recall¶

Accuracy measures how often the classifier makes the correct prediction. It’s the ratio of the number of correct predictions to the total number of predictions (the number of test data points).

Precision tells us what proportion of messages we classified as spam, actually were spam. It is a ratio of true positives(words classified as spam, and which are actually spam) to all positives(all words classified as spam, irrespective of whether that was the correct classificatio), in other words it is the ratio of

[True Positives/(True Positives + False Positives)]

Recall(sensitivity) tells us what proportion of messages that actually were spam were classified by us as spam. It is a ratio of true positives(words classified as spam, and which are actually spam) to all the words that were actually spam, in other words it is the ratio of

[True Positives/(True Positives + False Negatives)]

For classification problems that are skewed in their classification distributions like in our case, for example if we had a 100 text messages and only 2 were spam and the rest 98 weren't, accuracy by itself is not a very good metric. We could classify 90 messages as not spam(including the 2 that were spam but we classify them as not spam, hence they would be false negatives) and 10 as spam(all 10 false positives) and still get a reasonably good accuracy score. For such cases, precision and recall come in very handy. These two metrics can be combined to get the F1 score, which is weighted average(harmonic mean) of the precision and recall scores. This score can range from 0 to 1, with 1 being the best possible F1 score(we take the harmonic mean as we are dealing with ratios).

Answer:

The Random Forest is an ensemble Machine Learning model that is generally based on Decision Tree. Therefore, in order to explain it clearly, it is a good idea to first explain the Decision Tree.

  • Decision Tree:

The Decision Tree is a type of supervised learning algorithm, that is primarily used to deal with classification problem (although it is still a valid option for regression). The output for a Decision Tree algorithm is a tree consisting of nodes and branches (or edges). The tree starts from the root node (the node at top most level), where an input variable is read into the tree (for example, in this project, an input variable can be a one person's information, such as age, education, race, etc). Then the root node will split into several decision nodes, and each of these decision nodes will further split into more decision nodes, and so on so forth. Branches are used to connect a node with its sub-spliting nodes. Each decision node has a condition for spliting (for example, whether he is white or black). If a node cannot split anymore, then it is called terminal node, which contains a label (or classification result). The usage of a tree is quite simple, just read an input variable, and let it travel from the root node until it reaches a terminal node. The terminal node will indicate the classification result (such as this person is predicted to make more than 50k). Therefore it is quite straightforward to use a Decision Tree to make a prediction.

The training process is little more complicated compared with prediction. Due to the simple nature of Decision Tree, the most important thing here, is to decide the spliting condition for the nodes. There're many ways to train a Decision Tree, and one popular strategy is called "information-gain", which is what I am going to introduce below. Since a tree should start from the root node, we will need to find the spiliting condition for the root node first. Suppose the training data set has 10 features (such as sex, age, race, etc.), for each of these features we will calculate "if we use this feature to split the training dataset, how heterogeneous the spliting sets will be". If in one spliting set, almost all the persons earn more than 50k and in another split set, almost all the persons earn less than 50k, then this spliting approach that is highly heterogeneous and we say "information gain is huge using this feature to spliting". So starting from the training data set, we use the feature that will casue most "information gain" to create the root node, and then from the root node, use the same strategy to split it into more node, and so on. Finally, a boundary condition is needed when training a Decision Tree, in order to tell when the tree should stop spliting anymore (for example, when the number of terminal nodes has reached a max value which is defined earlier, we should stop spliting the tree). And this is one basic approach to train a Decision Tree.

  • Random Forest Reference:

Decision Tree is easy to use and actually not difficult to train. However, it has a drawback which is its high variance. We know the training data set is a sub set from the actual population, and the structure of a Decison Tree will change a lot if different samples (training data sets) are taken from the same population. In other words, when using different samples, we can get completely different result for predicting for a same person, whether or not he earns more than 50k.

This is somthing we want to mitigate, and Random Forest is such an approach to achieve this. The basic idea behind Random Forest is that "if one Decision Tree is not stable enough, how about generating multiple Decision Trees, and see the average result or let them voting?"

A Random Forest is actully a collection of several Decision Trees, and training a Random Forest is quite straightforward. Normally, we need to decide the number of Decesion Trees in the Random Forest. For example, let's say, we want a Random Forest consisting of 4 trees. So for 4 times, we extract a subset from the total training set, and we get subset_1, subset_2, subset_3, and subset_4. Each subset normally should account for 66% of the training set, and in each time, one subset is generated totally randomly, which is why we call it Random Forest. And, ok, so we have 4 subsets from the training set, and using each subset, we can generate a Decision Tree (using the approach I mentioned for training a Decision Tree). And we end up getting tree_1, tree_2, tree_3, and tree_4.

Now it is easy to use this Random Forest to make a prediction, that is, for each of 4 trees, we predict the classification result. So we will have 4 results totally. For example, tree_1 says ">50k", tree_2 says ">50k", tree_3 says "<=50k", and tree_4 says ">50k", we let all the results vote, and in the end, the final prediction is "This person earns more than 50k".
